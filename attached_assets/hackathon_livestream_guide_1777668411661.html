Is This Actually Possible?

✅ YES — 100% Possible

What you're describing — live streaming 10 teams on your own website, showing a real-time poll, and restricting poll access behind a participation code — is a well-understood technical pattern. Big platforms like Twitch, YouTube, and Zoom do variations of this at scale. You can build a lightweight version of this for a hackathon using modern, mostly free tools. The poll-gate (participation code) is simply an auth check before showing the vote UI. Let's break down every piece.

OBS Studio (on Presenter's PC)

Free software. The presenter installs OBS, selects their screen/camera, and OBS pushes the video to a streaming server using a protocol called RTMP (Real-Time Messaging Protocol). Think of RTMP as a live video "upload" pipe.

FREE RTMP Out
☁️

Streaming Service (CDN)

This receives the RTMP stream, encodes it into multiple quality levels, and distributes it globally via CDN. Options: YouTube Live (free, embedded), Mux (paid, more control), Cloudflare Stream (cheap).

YouTube = FREE Mux = Paid
📺

Video Player on Your Page

Your website shows the stream using an embed or an HLS.js player. For YouTube, it's a 1-line iframe embed. For Mux/Cloudflare, you get an HLS stream URL and use a player like Video.js or Plyr.

iframe HLS.js Video.js
💡 Simplest Approach for Hackathon Use YouTube Live (free) with a private/unlisted stream. Embed it on your website with an iframe. YouTube handles all the CDN, encoding, and bandwidth. You control who sees the embed by controlling access to your page via the participation code gate.
⚠️ One Important Note About YouTube Embedding If you use a YouTube "Unlisted" stream, anyone with the embed link can technically view it. To truly restrict, use a paid service (Mux, Cloudflare Stream) that gives you private HLS URLs, or use a signed URL approach. For a hackathon with trusted participants, YouTube Unlisted is perfectly fine.

Layer 2 — The Participation Code Gate

This is the "bouncer" of your system. It checks if a person is allowed to see the stream and vote. Here's exactly how it works technically.

What The Code Looks Like

Here's a simplified version of the backend verification logic:

Node.js / Express API // POST /api/verify-code app.post('/api/verify-code', async (req, res) => { const { code } = req.body; // 1. Look up code in database const record = await db.query( 'SELECT * FROM codes WHERE code = ?', [code] ); // 2. Check if it exists and hasn't been used if (!record || record.used) { return res.status(401).json({ valid: false }); } // 3. Mark as used (prevent reuse) await db.query( 'UPDATE codes SET used=true WHERE code=?', [code] ); // 4. Return a session token const token = jwt.sign({ code, role: 'participant' }, SECRET); res.json({ valid: true, token }); });

✅ What You Get

  • Only real participants can vote
  • Each code can only be used once
  • Each person can only vote once
  • You know exactly who voted (code = identity)
  • Non-participants see a blocked screen

⚠️ Things to Consider

  • If someone shares their code, another person can use it first
  • If user clears browser data, they lose their session
  • You need to securely store and distribute codes
  • Need a way to reset a code if someone loses it

Layer 3 — Real-Time Poll System

The poll lets authenticated participants vote for their favorite team, and all viewers see results update live as votes come in.

🔥 Firebase Realtime Database = Easiest Option Firebase is a Google service that acts as both your database AND your WebSocket. When you write a vote to Firebase, it automatically pushes the update to everyone who is "listening" on that data. No separate WebSocket server needed. Free tier is very generous and perfect for a hackathon.
Firebase Realtime Poll // When user votes — Frontend (React) const castVote = async (teamId) => { // Write vote to Firebase await firebase.database() .ref(`votes/${teamId}`) .transaction(count => (count || 0) + 1); // Mark this user as voted (prevent double-voting) await firebase.database() .ref(`voted/${participantCode}`) .set(true); }; // Listen for live updates — updates for ALL users automatically firebase.database().ref('votes') .on('value', snapshot => { const votes = snapshot.val(); // { teamAlpha: 23, teamBeta: 11 } setLiveResults(votes); // Updates React UI instantly });

Step-by-Step: How To Build It

The full implementation journey from zero to a working system.

01

Set Up Firebase Project

Go to firebase.google.com → Create Project → Enable Realtime Database and Authentication. Create your database with this structure:

{ "codes": { "HACK-A1B2": { "used": false, "name": "Rahul Sharma" }, "HACK-X9K2": { "used": false, "name": "Priya Mehta" } }, "votes": { "teamAlpha": 0, "teamBeta": 0 }, "voted": {} // tracks who already voted }
02

Build the Frontend (React / Next.js)

Create 2 pages: a Code Entry Page (landing page with input box) and a Watch Page (stream + poll, only accessible after verification). Use React Router or Next.js routing to protect the Watch page.

/ (root)
Code Entry Form
→ verified →
/watch
Stream + Poll
→ voted →
/results
See live chart
03

Code Verification Logic

On form submit, call Firebase to check if the code exists and hasn't been used. If valid, mark it as used and save a flag to localStorage (so refresh doesn't kick them out). Redirect to /watch.

04

Embed the Live Stream

On YouTube Live, start your stream. Copy the Video ID. On your /watch page, embed it. For 10 teams, you can switch the stream between teams or have multiple simultaneous streams (advanced).

// React component to embed YouTube stream function LiveStream({ videoId }) { return ( <iframe src={`https://www.youtube.com/embed/${videoId}?autoplay=1`} width="100%" height="500" allow="autoplay; encrypted-media" /> ); }
05

Build the Real-Time Poll

Display team buttons. On click, write the vote to Firebase. Use Firebase's onValue listener to watch the votes in real-time. Update a live bar chart using the incoming data. Use CSS transitions to animate the bar widths smoothly.

06

Generate & Distribute Codes

Write a simple script to generate unique codes (e.g. HACK-XXXX format) and bulk-upload them to Firebase. Email each participant their code. Keep a master sheet. Set up an admin route to add/reset codes if needed.

// Generate 200 unique codes const generateCode = () => { const chars = 'ABCDEFGHJKLMNPQRSTUVWXYZ23456789'; let code = 'HACK-'; for (let i = 0; i < 4; i++) { code += chars[Math.floor(Math.random() * chars.length)]; } return code; // e.g. "HACK-R7KP" };
07

Build Admin Dashboard

A protected admin page (password-only or Firebase Admin Auth) where you can: see live vote counts, see which codes are used, create a new poll question mid-event, and export results as CSV at the end.

08

Deploy to Vercel

Push your Next.js code to GitHub. Connect Vercel to your GitHub repo. It auto-deploys on every push. Set your Firebase config as environment variables in Vercel. Your site is live at yourproject.vercel.app or a custom domain.

Real Challenges & How to Handle Them

Things that can go wrong and what to do about them — so you're not caught off-guard.

📤

Code Sharing / Leaking

Someone shares their code in a WhatsApp group. Others use it before the intended person. Solution: Mark codes as used the moment they're entered. First person to enter wins. Brief participants that codes are single-use and personal.

🔄

Page Refresh Loses Session

User refreshes and gets sent back to code entry page. Solution: Store a token in localStorage after verification. On page load, check for existing token first before showing code entry.

📶

Stream Latency / Buffering

YouTube Live has ~10-20 second delay. Solution: Acceptable for a hackathon. If lower latency needed, use Mux or Cloudflare Stream (2–5 second delay). Tell participants to expect a slight delay.

🗳️

Double Voting

What if the same person votes twice by opening different tabs? Solution: Use the participation code as the unique voter ID. Once they vote, write their code to the "voted" list in Firebase. Check this before accepting any vote.

📱

Mobile Responsiveness

Many participants will join on phones. Solution: Build mobile-first. Use Tailwind responsive classes. Test on Chrome mobile. YouTube embed is already responsive with proper CSS aspect-ratio handling.

🔇

Stream Goes Down Mid-Event

Presenter's internet drops, OBS crashes. Solution: Have a backup — Zoom meeting ID as fallback. The poll system is independent and keeps working even if stream is down.

It's a combination of 3 well-understood systems: live streaming (solved by YouTube/Mux), auth gating (solved by a simple DB code check), and real-time polls (solved by Firebase). All three are well-documented, have free tiers, and can be built in 1-2 weeks by a developer with basic web skills. The key is to not build it from scratch — use Firebase for realtime, YouTube for streaming, and Vercel for hosting.
